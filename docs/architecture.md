# Senda вҖ” Architecture

This document describes the runtime architecture of the Senda desktop app and
how its Rust crates fit together. The product vision and roadmap live in
[`PLAN.md`](../PLAN.md) (at the repository root of the parent `senda` project);
this file is the engineer-facing complement.

* **apps/desktop/src** вҖ” SolidJS + TypeScript UI. Talks to the backend
  exclusively through Tauri's `invoke` and event channels. No business logic
  lives in the browser.
* **apps/desktop/src-tauri** вҖ” Tauri shell. Owns the window, registers IPC
  commands, and wires the Rust crates. Phase 0 only ships a `hello_world`
  command as an IPC smoke test.
* **crates/core** вҖ” Pure data types: `Agent`, `Repo`, `Automation`, etc.
  Every other crate depends on this one. No filesystem, no network.
* **crates/acp-client** вҖ” Async client for the Agent Client Protocol. Designed
  to be publishable to crates.io independently of Senda. See its README for
  details on the wire format we implement.
* **crates/git-providers** вҖ” `RepoProvider` trait with GitHub, Azure and
  Generic implementations. Phase 0 ships URL parsing + tests; the network
  surface (clone/pull/push/PR) lands in Phase 2.
* **crates/agent-parser** вҖ” Canonical agent format reader/writer plus the
  three transpilers. The canonical doc is the source of truth; generated CLI
  files are artefacts. Warnings are emitted (not errors) when a target CLI
  cannot represent a field.
* **crates/scheduler** вҖ” Public API surface for Phase 3. Cron + event +
  webhook + manual triggers, all guarded by idempotency, rate limiting and
  backpressure policy. Phase 0 ships only the registry.

## IPC and types

Phase 0 hand-writes the TypeScript IPC types in
`packages/shared-types/src/index.ts`. Phase 1 will switch this to
[`tauri-specta`](https://github.com/specta-rs/tauri-specta), which derives
TypeScript bindings directly from `#[derive(specta::Type)]` Rust structs.
Every type in `crates/core` already derives `specta::Type` so the migration
will be a build.rs change, not a refactor.

## Filesystem layout (runtime)

Owned by Senda (canonical):

```
~/.senda/
в”ңв”Җв”Җ agents/             canonical agent documents (source of truth)
в”ңв”Җв”Җ drafts/             agents in progress
в”ңв”Җв”Җ cache/              clones of connected git repositories
в”ңв”Җв”Җ data.db             SQLite (automations, runs, telemetry, warnings)
в””в”Җв”Җ logs/
```

Generated by Senda from the canonical docs (artefacts only):

```
~/.copilot/agents/{name}.agent.md
~/.claude/agents/{name}.md
~/.gemini/agents/{name}.toml
```

If pre-existing files in those folders are not registered in
`~/.senda/manifest.json`, Senda treats them as **external** read-only with an
"Import to Senda" action.

## Why these boundaries?

* The desktop app is a thin shell over the crates. If we ever need to
  ship a headless `senda` CLI, the same crates stand alone.
* `senda-acp-client` is fully decoupled from Senda concerns and ships its own
  README so it can be published as `acp-client-rs` (or similar) without
  carrying any product code.
* `senda-agent-parser` keeps every transpiler behind a `Transpiler` trait so
  new CLIs can be added without touching the core.
